# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

OpenMenses is an offline-first, privacy-first menstrual cycle tracking app. All data stays on-device — no servers, no telemetry, no cloud. The architecture is a monorepo: Proto definitions → Generated code → Go engine ↔ TypeScript UI → Mobile wrappers (iOS/Android).

## Build Commands

All commands run from the repo root via Make:

```bash
make proto-lint          # Lint proto files with buf
make proto-generate      # Regenerate Go + TS code from proto
make proto-breaking      # Check for breaking proto changes vs main
make engine-lint         # golangci-lint on Go engine
make engine-test         # Go tests
make ui-lint             # ESLint + typecheck on TS UI
make ui-test             # Vitest
make lint                # All linters
make test                # All tests
make ci                  # Full CI: lint + generation check + tests
make mobile-setup        # Install gomobile tooling (one-time)
make mobile-ios          # Build Engine.xcframework via gomobile bind
make ui-bundle           # Build UI production bundle into ui/dist/
```

Run a single Go test:

```bash
cd engine && go test ./internal/rules/ -run TestDetectCycles
```

Run TS tests:

```bash
cd ui && npm test
```

## Architecture

| Layer         | Location               | Role                                                   |
| ------------- | ---------------------- | ------------------------------------------------------ |
| **Proto**     | `proto/openmenses/v1/` | Canonical contract — all types and RPCs defined here   |
| **Generated** | `gen/go/`, `gen/ts/`   | Auto-generated by buf — **never hand-edit**            |
| **Engine**    | `engine/`              | Go domain logic, storage, validation, service handlers |
| **UI**        | `ui/`                  | TypeScript presentation layer — **no domain logic**    |
| **Mobile**    | `mobile/`              | Thin native shells (iOS/Android) — no domain logic     |

### iOS CI (Xcode Cloud)

Xcode Cloud runs `mobile/ios/ci_scripts/ci_post_clone.sh` after cloning. This script installs Go and Node.js, builds `Engine.xcframework` via `make mobile-ios`, and runs `make ui-bundle`. The framework is gitignored (166 MB build artifact) — it must be built in CI, not committed.

### Engine internals (`engine/internal/`)

- **service/** — Connect-RPC handlers implementing `CycleTrackerService`, split by resource (profiles, observations, medications, cycles, exports)
- **storage/** — `Repository` interface with two backends: `memory/` (tests) and `sqlite/` (on-device persistence)
- **rules/** — Domain algorithms: cycle detection, phase estimation, statistics
- **validation/** — Two-layer: Protovalidate (schema) + custom Go (domain)
- **timeline/** — Timeline construction and queries
- **insights/** and **predictions/** — Insight/prediction generation

Public API surface: `engine/pkg/openmenses/engine.go` — `Engine` type with functional options (`WithSQLite`, `WithInMemory`).

## Key Design Constraints

- **Resource-oriented design** (AIP-compliant): design around nouns, use standard CRUD methods, no custom verbs. See `Design_Guidelines.md`.
- **Proto-first workflow**: define/update `.proto` → lint → generate → implement in Go → test. See `AGENT.md` §3 for mandatory workflow.
- **Offline-first**: no network calls required for core functionality.
- **Privacy**: no telemetry, analytics, or remote logging.
- **FieldMask** required for Update operations; **pagination** required for List operations.
- **No custom verbs**: model state changes as Update operations on resource fields (e.g., `UpdateSubscription` with `state=CANCELLED`, not `CancelSubscription`).
- **Names over IDs**: request messages use `string name` fields, not `int64 id`.

## Code Style

**Go** (`engine/.golangci.yml`): gofmt with simplify, goimports with local prefix `github.com/2ajoyce/openmenses`. Table-driven tests. Max ~1000 lines per package before splitting. Domain-oriented packages, not technical layers. Small interfaces defined where used. See `Go_Code_Guidelines.md`.

**TypeScript**: ESLint + strict TypeScript. Vitest for testing. All custom CSS in a single `ui/src/app/theme.css` file using `--om-` design tokens and `om-` utility classes. No CSS-in-JS, no CSS modules, no Tailwind, no inline styles. See `UI_Design_Guidelines.md`.

**Proto**: buf lint with STANDARD rules. Commit generated code alongside proto changes. Every enum must have a `0` value named `<TYPE>_UNSPECIFIED`. Use `google.api.resource` and `google.api.http` annotations.

## Workflow for New Features

Follow the mandatory agent workflow from `AGENT.md` §3:

1. Identify the resource and parent-child hierarchy
2. Choose standard lifecycle methods (Get, List, Create, Update, Delete) before custom actions
3. Draft/update `.proto` in `proto/openmenses/v1/`
4. `make proto-lint && make proto-generate`
5. Implement in `engine/internal/` (service, storage, rules as needed)
6. `make engine-lint && make engine-test`
7. Commit proto changes and generated code together
8. Check for breaking changes: `make proto-breaking`
