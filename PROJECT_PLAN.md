# Cycle Tracker Project Plan

## Project Goals

Create an open-source menstrual and cycle tracking application with the following principles:

- Offline-first operation
- No central server
- User-owned data
- Transparent algorithms
- Cross-platform deployment
- Fully open data model

The application should prioritize scientific transparency and avoid assumptions about user identity or reproductive goals.

The system will separate domain logic from the user interface and platform integrations.

---

# Core Architecture

The application consists of four primary layers.

## 1. Proto Schema

The proto schema defines the canonical data model and service contracts.

It serves as the single source of truth for:

- entities
- enums
- data relationships
- validation rules

The proto schema includes validation annotations using Protovalidate.

The schema defines:

User-entered records

- BleedingObservation
- SymptomObservation
- MoodObservation
- Medication
- MedicationEvent

System-derived records

- Cycle
- PhaseEstimate

System-generated interpretations

- Prediction
- Insight

Proto definitions are language-neutral and will generate bindings for both Go and TypeScript.

---

## 2. Domain Engine (Go)

The Go domain engine implements the core application logic.

Responsibilities include:

- cycle detection
- prediction algorithms
- symptom pattern analysis
- validation enforcement
- domain rule enforcement
- data import/export
- migration logic

The domain engine should be fully testable independently of the UI.

The Go layer is considered the authoritative implementation of domain behavior.

---

## 3. TypeScript UI

The user interface is implemented using TypeScript.

Responsibilities include:

- onboarding flows
- logging interactions
- calendar views
- visualization and charts
- accessibility
- local interaction logic

The UI should not contain domain rules.

All domain logic should remain in the Go engine.

---

## 4. Native Platform Shell

A thin native wrapper provides device integration.

Responsibilities include:

- HealthKit access
- local notifications
- platform backup integration
- secure storage
- device permissions

The shell hosts the runtime and exposes platform APIs to the application.

### Build and packaging

The Go engine compiles to native libraries via `gomobile bind`: `.aar` for Android, `.xcframework` for iOS. See https://go.dev/wiki/Mobile#sdk-applications-and-generating-bindings for reference.

### WebView architecture

The native shell hosts a WebView that renders the React UI. On startup, the shell initializes the Go engine in-process and binds a localhost-only Connect-RPC HTTP listener. The WebView calls this listener using standard Connect-RPC HTTP requests, providing a platform-agnostic bridge between the UI and the engine.

### Platform-specific integrations

Platform-specific integrations (HealthKit on iOS, Google Fit on Android) are implemented in Swift or Kotlin within the native shell. They are not implemented in Go or TypeScript. This keeps the Go engine platform-neutral and avoids pulling platform SDKs into the shared codebase.

### Backups

iCloud and platform backups work automatically because all persistent data is stored in a local SQLite file. The operating system's built-in backup mechanism (iCloud on iOS, device backup on Android) includes this file without any custom integration.

# Data Model Philosophy

The domain model separates three categories of information.

User-entered data

- raw observations recorded by the user

System-derived data

- interpretations derived from user data

Predictions

- forward-looking estimates generated by algorithms

Insights

- historical pattern summaries derived from analysis

This separation prevents the system from confusing observed facts with predictions.

---

# Validation Strategy

Validation is implemented in two layers.

Schema validation (Proto)

Enforced using Protovalidate.

Examples include:

- string length limits
- enum restrictions
- list cardinality
- required fields
- basic structural rules

Domain validation (Go)

Enforced inside the domain engine.

Examples include:

- cycle boundary rules
- prediction eligibility
- algorithm constraints
- cross-record consistency

---

# Offline Operation

The application is designed to operate without a central server.

Data storage occurs locally on the device.

Backups occur using platform-native mechanisms such as:

- iCloud backups
- device backup systems
- user export

The system does not rely on cloud services operated by the project.

---

# Privacy Model

The system prioritizes user privacy.

Design principles include:

- no mandatory account
- no telemetry by default
- no centralized data storage
- transparent algorithms
- exportable raw data

Users maintain full control of their information.

# Guiding Principles

1. Domain logic must remain independent of UI frameworks.
2. Observations must always remain distinguishable from predictions.
3. The data model must remain stable and versionable.
4. Algorithms must be transparent and explainable.
5. The application must operate without centralized infrastructure.

All analytic and predictive algorithms are documented in [docs/algorithms.md](docs/algorithms.md) for user and contributor transparency.
